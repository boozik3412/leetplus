# LeetPlus

LeetPlus - SaaS-система для управления ассортиментом компьютерных клубов и сетей клубов. Приложение собирает данные из LAngame, приводит товары разных клубов к единой сетевой картине и помогает управлять продажами, остатками, дефицитом, маржинальностью и качеством ассортимента. Следующее стратегическое направление - отдельный модуль `Гости` для аналитики клиентской базы, бонусов, CRM, лояльности, коммуникаций и управления гостевым потоком.

## Текущий статус

На 20.05.2026 проект находится в production-стадии раннего аналитического продукта: рабочий контур развернут на VDS, подключен к реальным источникам LAngame и развивается быстрыми итерациями от пользовательских сценариев. Модуль `Гости` перешел из проектирования в первый production read-only слой: есть data foundation, дашборд, полный отчет, карточка гостя, базовый mini CRM, первый отчет `Контроль персонала`, расчет загрузки по PC capacity и отдельный раздел общей синхронизации.

- Web: `https://leetplus.ru`
- API: `https://api.leetplus.ru`
- Production-ветка: `main`
- VDS: Ubuntu 24.04 LTS
- Автообновление VDS: включено, сервер проверяет `origin/main` через `leetplus-deploy.timer`
- SSL: Let's Encrypt для `leetplus.ru`, `www.leetplus.ru`, `api.leetplus.ru`
- Production-почта: `reports@leetplus.ru` через Mail.ru/VK WorkSpace SMTP
- Основной рабочий процесс: локально меняем код, проверяем, коммитим и пушим в `main`; VDS забирает изменения автоматически
- Локальную БД и локальные сервисы не трогаем без отдельной необходимости: актуальная проверка пользовательского результата идет сразу на `leetplus.ru`

Подключенные production-источники LAngame:

- `1337.langame.ru`

- `46.langamepro.ru`

## Что уже реализовано

### Дашборд

- KPI по выручке, прибыли с потерями, списаниям, остаткам и риску `out-of-stock`.
- Компактные фильтры периода и клубов в верхнем блоке дашборда.
- Автоматическое применение фильтров сразу после выбора, без отдельной кнопки `Применить`.
- Быстрый запуск синхронизации из дашборда.
- Динамика продаж по 8 отрезкам выбранного периода: деньги, штуки и изменение к предыдущему отрезку.
- Для произвольного периода дашборд сравнивает 8 аналогичных периодов, где выбранный период является последним отрезком.
- Формат месяцев в графиках приведен к короткому виду вроде `янв.26`.
- `SKU без движения` с переключателями 7, 14 и 21 день.
- `OOS, SKU` и карточка остатков менее 3 дней продаж.
- `Активный ассортимент`: активными считаются SKU с текущим остатком либо с продажами за последние 14 дней.
- ТОП-10 SKU по выручке с сетевой группировкой через канонические товары.
- Переключатель группировки ТОП SKU перенесен внутрь блока ТОП-10, чтобы не выглядеть глобальным фильтром.
- Аналитика категорий: доли категорий в выручке/прибыли и показатели эффективности категории.
- Адаптация шапки и ключевых KPI под узкие экраны.
- Управленческая перекомпоновка первого экрана вокруг коммерческой картины сети.
- Сводный executive dashboard объединяет общую выручку, гостей, ассортимент и игровую загрузку сети.
- Загрузка считается в процентах как отыгранные часы / возможные PC-часы за выбранный период; количество ПК берется из LAngame `global/types_of_pc_in_clubs/list` и `global/linking_pc_by_type/list`.
- Блоки `Главное внимание`, `Что сделать сегодня` и `Что изменилось` с прямыми переходами к отчетам.
- Выбор текущих и полных периодов для корректного сравнения период к периоду.

- `Остатки и потребность`.
- ABC-анализ с переключением между выручкой и прибылью.
- ТОП SKU и SKU с низкой маржой.
- `Новинки`: товары, у которых первый в истории положительный остаток появился за последние 90 дней.
- LFL год к году по выручке, прибыли и штукам.

- `Риск out-of-stock` дополнен коммерческой оценкой выручки и прибыли в риске за день, а также прибыли в риске за выбранный период.
- `Деньги в риске`: гибридный отчет, который объединяет прибыль в риске из-за OOS и деньги, замороженные в товарах без движения.

- Сводные экспорты используют русские названия листов и параметров.
- Раздел отчетов собран в раскрывающиеся плашки, чтобы быстрее находить нужный отчет.
- Полные табличные страницы для ключевых отчетов.

### Утилиты парсинга товаров

- Умный анализ похожих товаров между клубами.
- Создание канонических SKU для сетевой отчетности.
- Подтверждение и отклонение найденных групп.

## Правила синхронизации и истории

Эти правила важны, потому что отчеты строятся на исторических фактах продаж и остатков.

- Повторная синхронизация должна быть идемпотентной: уже загруженные операции обновляются/сопоставляются по внешним идентификаторам, а не создают дубли.
- История продаж не должна ломаться при переименовании товара: в фактах продаж хранится снимок названия на момент продажи.
- Если товар пропал из номенклатуры LAngame, исторические продажи сохраняются; сам товар может быть помечен как отсутствующий/неактивный через служебные поля.
- Синхронизация из LAngame не должна сама проставлять `canonicalProductId`.
- Объединение товаров в канонический SKU выполняется только через утилиту анализа и ручное подтверждение.
- Любые изменения, влияющие на историю, должны сохранять внешние ID, snapshot-поля и связь с tenant/store.
- Будущая история гостей не должна ломаться при изменении профиля гостя в LAngame.
- Синхронизация гостей не должна перетирать ручные CRM-статусы, заметки, сегменты, согласия на коммуникации и другие данные, созданные в LeetPlus.
- Персональные данные гостей считаются чувствительными: телефон, email, ФИО, дата рождения и документы требуют отдельного контроля доступа и осторожного хранения.
- Для коммуникаций с гостями нужны согласия по каналам, история согласий и возможность отписки.

## LAngame

Подключение выполняется через `/settings`:

1. Указать API-ключ.
2. Добавить домены клубов.
3. Сохранить настройки.
4. Запустить синхронизацию.

Домены клубов вводятся одним из двух способов:

- через запятую с пробелом: `1337.langame.ru, 443.langame.ru`;
- каждый домен с новой строки.

В доменах не нужно указывать `https://` и путь `/public_api`; приложение нормализует домен и само формирует базовый URL вида `https://<domain>/public_api`.

Ключ сохраняется на backend в зашифрованном виде и привязывается к текущему `tenant`.

Базовый URL домена:

```text
https://<domain>/public_api
```

Заголовок авторизации:

```text
X-API-KEY
```

Ключевые методы, используемые интеграцией:

- `GET /clubs/list`
- `GET /products/list`
- `GET /goods/list?club_id=...`
- `GET /products/expense?page_limit=&page=&date_from=&date_to=`
- `GET /products/arrival?page_limit=&page=`
- `GET /transactions/list`
- `GET /log_cash_transaction/list`
- `GET /all_operations_log/list`

Если один из вспомогательных endpoints LAngame падает, sync job может быть помечен как `FAILED`, даже если уже полученные товары, остатки и продажи сохранены. Это ожидаемое поведение внешней интеграции: статус job отражает полный проход, а не факт сохранения каждой отдельной сущности.

### Плановая синхронизация

Для cron и сервисных вызовов предусмотрен endpoint:

```text
POST /integrations/langame/scheduled/sync
Header: x-sync-service-token: <SYNC_SERVICE_TOKEN>
```

Поддерживаемые режимы:

- `QUICK`
- `INVENTORY`
- `CATALOG`
- `BACKFILL`
- `FULL`

Пример вызова:

```powershell
curl -X POST "https://api.example.ru/integrations/langame/scheduled/sync" `
  -H "x-sync-service-token: <SYNC_SERVICE_TOKEN>" `
  -H "Content-Type: application/json" `
  -d "{\"mode\":\"QUICK\"}"
```

## Модель доступа

`Tenant` - сеть клубов. Пользователь принадлежит одному tenant и видит только данные своей сети.

`Store` - конкретный клуб внутри сети.

Для интеграции по клубам сохраняются внешние идентификаторы и домены, а tenant-уровневые ключи лежат в зашифрованном виде.

## Важные правила

- Не коммитить `.env` и реальные API-ключи.
- Не менять Prisma на `v7` без отдельного решения.
- Не поднимать вторую PostgreSQL в Docker локально без явной необходимости.
- Не менять `APP_ENCRYPTION_KEY` без процедуры ротации.
- Не делать автоматическое объединение товаров во время LAngame sync.
- Не удалять исторические факты продаж при переименовании или исчезновении товара из внешней номенклатуры.
- Любые изменения в расчетах прибыли, маржи, ABC и LFL должны явно учитывать источник себестоимости.
