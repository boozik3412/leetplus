# LeetPlus

LeetPlus - SaaS-система для управления ассортиментом компьютерных клубов и сетей клубов. Приложение собирает данные из Langame, приводит товары разных клубов к единой сетевой картине и помогает управлять продажами, остатками, дефицитом, маржинальностью и качеством ассортимента. Следующее стратегическое направление - отдельный модуль `Гости` для аналитики клиентской базы, бонусов, CRM, лояльности, коммуникаций и управления гостевым потоком.

## Текущий статус

На 29.05.2026 проект находится в production-стадии раннего аналитического продукта: рабочий контур развернут на VDS, подключен к реальным источникам Langame и развивается быстрыми итерациями от пользовательских сценариев. LeetPlus расширяется в операционную систему для сети компьютерных клубов: есть дашборд сети, гости/CRM, ассортимент, маркетинг и начат отдельный блок `Персонал`.

### Telegram auth webhook replies

Для основного входа в геймификацию через Telegram `/guest-portal/telegram/webhook` по умолчанию возвращает безопасный `reply` payload для внешнего bot-adapter. Если нужно, чтобы LeetPlus API сам отправлял кнопку `request_contact` и финальное сообщение в Telegram, на VDS включается `GUEST_GAME_TELEGRAM_WEBHOOK_REPLY_ENABLED=true` и задается `GUEST_GAME_TELEGRAM_WEBHOOK_REPLY_BOT_TOKEN` или общий `GUEST_GAME_TELEGRAM_BOT_TOKEN`. Raw `chat_id` используется только из текущего Telegram update в памяти, в ответе API остается только маска, статус `replyDispatch` и недостающие env; raw update, bot token, телефон и Langame payload не сохраняются и не отдаются на frontend.

Guest Game Hub на `/guests/gamification` показывает для этого отдельный readiness-пункт `Telegram reply sender для входа`: видны только статусы webhook secret, env-флага sender, bot token, required env и следующий QA-шаг; сами секреты, raw chat id и raw Telegram update не выводятся.

### Реферальная регистрация в геймификации

`/play/game` выдает участнику геймификации непрозрачный HMAC-код `lp_ref_*` и ссылку вида `/play?clubId=<tenant:club>&ref=<code>`. `/play` принимает этот `ref`, показывает безопасный баннер приглашения и передает код только в момент успешного завершения авторизации: OTP/SMS-код, Telegram-бот с contact-share или звонок пользователя на номер.

Backend записывает факт регистрации по ссылке как `GuestGameEvent` с `eventType=GAME_REFERRAL_ACCEPTED` и `source=GUEST_PORTAL_REFERRAL`. Пригласивший находится пересчетом HMAC-кода по активным `GuestGameProfile` выбранного tenant/store; в payload сохраняются `inviterProfileId`, `inviterGuestId`, `valid`, `selfReferral`, `eligibleForReward`, `clubId` и маска кода. Публичная регистрация по-прежнему не создает общий `Guest`, не сохраняет raw phone, токены или Langame payload и не смешивает игроков с общей базой гостей до обычной snapshot-синхронизации.

Regression-тесты `guest-portal.service.spec.ts` закрепляют этот инвариант для всех рабочих каналов входа в `/play`: Telegram-бота, звонка пользователя на номер, входящего звонка с последними 4 цифрами и OTP/SMS.

Guest Game Hub читает такие eligible-события как snapshot-fact `GUEST_GAME_REFERRAL` с `eventType=REFERRAL_ACCEPTED`: в batch pipeline проходят только `valid=true`, `eligibleForReward=true` и не self-referral, а обработка идет по `inviterProfileId`. Это позволяет включить XP/бонус пригласившему обычным правилом Guest Game Hub без участия админа и без создания общего `Guest`; если профиль еще не связан с Langame-гостем, награда остается на игровом профиле до дальнейшей связки.

### Langame balance write API

Для геймификации подтвержден отдельный master endpoint Langame: `POST /master_api/guests/balance/phone`. Backend вызывает его по активному tenant-источнику Langame с заголовком `X-Request-Token`, используя сохраненный зашифрованный ключ источника; токены не передаются на frontend и не пишутся в audit.

Payload: `phone`, `type`, `sum`, `comment`, где `type` может быть `bonus_balance` для бонусов или `balance` для денежного баланса. Положительный `sum` начисляет, отрицательный списывает. Guest Game Hub использует этот endpoint через bonus ledger: награды без ручного подтверждения становятся `APPROVED`, попадают в очередь и отправляются по расшифрованному телефону гостя; в ledger metadata и audit сохраняется только маска телефона. Бонусные reward types по умолчанию уходят как `bonus_balance`, а денежные reward types (`BALANCE`, `DEPOSIT` и родственные) можно явно включить через `rewardTypes`/env и они уходят как `balance`; оба пути закреплены API-тестами. Signed-операции разделены в локальном учете: отрицательный `bonus_balance` уменьшает `GuestBonusBalanceCurrent`, а `balance` фиксируется в ledger/audit и не меняет бонусный current-контур до появления отдельного денежного current-баланса.

Боевой dispatcher включается env-флагом `LANGAME_BONUS_ACCRUAL_ENABLED=true`; путь по умолчанию: `LANGAME_BONUS_ACCRUAL_PATH=/master_api/guests/balance/phone`.

Для автономной работы без админского запуска добавлен API-side scheduler `GuestBonusLedgerSchedulerService`: в production он стартует при наличии `SYNC_SERVICE_TOKEN`, если `GUEST_GAME_BONUS_LEDGER_SCHEDULER_ENABLED` не выключен явно. Scheduler вызывает существующий `POST /guests/gamification/scheduled/bonus-ledger/dispatch`-контур внутри backend, пропускает пересекающиеся тики и логирует только агрегаты tenant/queued/confirmed/failed/blocked без raw phone, токенов и Langame payload. Настройки: `GUEST_GAME_BONUS_LEDGER_SCHEDULER_INTERVAL_MS`, `GUEST_GAME_BONUS_LEDGER_SCHEDULER_LIMIT`, `GUEST_GAME_BONUS_LEDGER_SCHEDULER_DRY_RUN`, `GUEST_GAME_BONUS_LEDGER_SCHEDULER_QUEUE_APPROVED_REWARDS`, `GUEST_GAME_BONUS_LEDGER_SCHEDULER_TENANT_ID`, `GUEST_GAME_BONUS_LEDGER_SCHEDULER_TENANT_SLUG`, `GUEST_GAME_BONUS_LEDGER_SCHEDULER_REWARD_TYPES`. Реальная запись в Langame все равно требует `LANGAME_BONUS_ACCRUAL_ENABLED=true`; если write-флаг не включен, ledger остается в safe dry-run/disabled режиме.

Для внешнего Telegram/MAX bot-consumer добавлены сервисные endpoints под тем же `SYNC_SERVICE_TOKEN`: `POST /guests/gamification/scheduled/deliveries/bot/pull` принимает `tenantId` или `tenantSlug`, `channels` и `limit`, возвращает только готовые `READY_FOR_BOT` доставки из `GuestGameDelivery` с безопасным текстом сообщения, масками и подтвержденным bot identity для отправки; `POST /guests/gamification/scheduled/deliveries/bot/ack` принимает `deliveryId`, `status=SENT|FAILED|BLOCKED`, короткие provider-поля и переводит delivery в итоговый статус с записью `GuestGameDeliveryEvent`. Повторный terminal ack с тем же статусом идемпотентно возвращает текущую delivery без нового события, а смена terminal-статуса требует явного retry из Guest Game Hub. Эти endpoints не делают live-запросы в Langame, не пишут в Langame, не отдают raw phone и не сохраняют raw Telegram update.

Для VDS добавлен runner внешнего Telegram consumer: после `api build` его можно запускать командой `pnpm --filter api guest-game:bot-consumer`. Настройки: `GUEST_GAME_BOT_CONSUMER_API_URL` (fallback `API_URL`), `GUEST_GAME_BOT_CONSUMER_SYNC_TOKEN` (fallback `SYNC_SERVICE_TOKEN`), `GUEST_GAME_BOT_CONSUMER_TENANT_ID` или `GUEST_GAME_BOT_CONSUMER_TENANT_SLUG`, `GUEST_GAME_BOT_CONSUMER_CHANNELS`, `GUEST_GAME_BOT_CONSUMER_LIMIT`, `GUEST_GAME_BOT_CONSUMER_DRY_RUN`, `GUEST_GAME_BOT_CONSUMER_INTERVAL_MS`, `GUEST_GAME_BOT_CONSUMER_MAX_TICKS`, `GUEST_GAME_BOT_CONSUMER_REQUEST_TIMEOUT_MS`, `GUEST_GAME_BOT_CONSUMER_TELEGRAM_BOT_TOKEN` (fallback `GUEST_GAME_TELEGRAM_BOT_TOKEN`). По умолчанию runner работает в dry-run: pull выполняется, внешняя отправка и ack не выполняются. Реальная Telegram-отправка требует `GUEST_GAME_BOT_CONSUMER_DRY_RUN=false` и bot token; MAX в runner оставлен только для dry-run до подтвержденного API-контракта. В итоговом логе runner выводит `acked` и `idempotentAcks`, чтобы в journal было видно безопасное повторное подтверждение после ретрая или сетевого дубля.

Готовые VDS-шаблоны для этого runner лежат в `docs/deployment/systemd`: env-example без секретов, one-shot `leetplus-guest-game-bot-consumer.service`, periodic `leetplus-guest-game-bot-consumer.timer` и runbook dry-run -> timer -> canary real-send. Реальный `/etc/leetplus/guest-game-bot-consumer.env` хранится только на VDS; шаблоны по умолчанию оставляют `GUEST_GAME_BOT_CONSUMER_DRY_RUN=true`, `GUEST_GAME_BOT_CONSUMER_MAX_TICKS=1` и канал `TELEGRAM`.

В Guest Game Hub блок `Outbox выдачи` дополнительно показывает `VDS bot-consumer`: API-visible режим runner, pending `READY_FOR_BOT` по Telegram/MAX, счетчики сохраненных ack `SENT/FAILED/BLOCKED`, последний ack, required env, следующий безопасный шаг и ссылку на runbook `docs/deployment/systemd`. Это наблюдаемость по сохраненным delivery/event и env-именам без токенов, raw phone, chat id, raw Telegram update или Langame payload.

Повторная подготовка outbox безопасна для production lifecycle: LeetPlus переоценивает старые `BLOCKED`-снимки, например после нового Telegram contact-share или согласия на игровом профиле, но не переводит terminal-доставки `SENT`, `FAILED` и `CANCELED` обратно в `READY`. Для повторной попытки failed-доставки нужен явный ручной перевод статуса, а не скрытый refresh: оператор может вернуть `FAILED` в `READY` только если сохраненная готовность по-прежнему `READY_FOR_BOT` или `READY_FOR_CASHIER`, при этом старый `failedAt` очищается, а `SENT` и `CANCELED` нельзя воскресить вручную.

Guest Game Hub показывает scheduler как отдельный пункт readiness `BONUS_LEDGER_SCHEDULER`: статус `READY` появляется только когда автономный scheduler включен, не форсирован в dry-run и одновременно разрешен `LANGAME_BONUS_ACCRUAL_ENABLED=true`. Этот пункт также выводит in-memory runtime-детали последнего scheduler-run: состояние, интервал, scope, reward types, последний запуск, агрегированный результат и последний overlap-skip без raw phone, токенов и Langame payload. Пилотный чек-лист `Bonus ledger -> Langame` теперь отличает ручной dispatch от настоящего автоначисления без админского участия.

Публичная регистрация участника геймификации доступна на `/play` без сотруднической авторизации. Страница загружает клубы с включенным `Store.gamificationEnabled` или активными правилами Guest Game Hub, поддерживает поиск по городу/клубу/адресу, карту-схему выбора клуба, геолокационную сортировку и фильтр радиуса при заполненных `Store.latitude/longitude`. Для сетевого каталога и QA-ссылок `/play` также принимает deep-link параметры `clubId`, `club` или `storeId` и сразу выбирает найденный клуб без нового API-контракта. Перед OTP гость подтверждает участие в квестах и обработку телефона для игрового профиля, бонусов и безопасной сверки с Langame; факт согласия сохраняется как безопасный `GuestGameEvent`. После OTP LeetPlus создает или переиспользует отдельный `GuestGameProfile` по подтвержденному телефону и tenant scope, записывает `profileId` в гостевую сессию и не создает запись в общем `Guest`. При повторном входе `/play` проверяет текущий guest-token через `GET /guest-portal/session/game-summary` и показывает блок активной игры с текущим профилем, клубом, XP, бонусами и быстрыми переходами в `/play/game` и кабинет клуба; без сессии остается обычный выбор клуба и OTP. Если кабинет открыт в статусе `GAME_PROFILE`, гость видит отдельный onboarding без предупреждения о ненайденном общем госте: профиль, связь с Langame, первый квест, канал наград и чекин ведут к существующим блокам публичного кабинета. Когда синхронизированный Langame-гость уже есть или появляется после обычной guest foundation sync, LeetPlus безопасно связывает его с `GuestGameProfile` по подтвержденному `phoneHash`, пишет `GAME_PROFILE_LINKED` без raw phone и протягивает `guestId` в игровые награды, события, delivery и bonus ledger; audit payload события и ответ ручной проверки показывают только безопасные счетчики backfill по этим контурам. В публичном кабинете блок лояльности берет бонусы из `GuestBonusBalanceCurrent` с fallback на последний `GuestBonusBalanceSnapshot`, поэтому ledger-начисление видно гостю до следующего исторического snapshot. Рядом с кошельком наград публичный кабинет показывает безопасную историю `GuestBonusLedgerEntry`: сумму, гостевой статус, источник квеста/лутбокса/Battle Pass, клуб, дату и баланс после подтверждения без raw phone, токенов, `langameRequest`, `langameResponse` и технических ошибок Langame.

`GuestGameProfile` хранит собственное игровое коммуникационное согласие: статус, источник, время согласия и время отписки. OTP-регистрация с принятым игровым согласием и Telegram contact-share записывают `GRANTED` на отдельный игровой профиль, поэтому новый участник может стать готовым к Telegram/MAX outbox без создания записи в общем `Guest`. Если позже профиль связывается с Langame guest или CRM lead, Guest Game Hub берет последнее явное состояние согласия между игровым профилем, `Guest` и lead, чтобы новая отписка не игнорировалась, а свежий Telegram contact-share мог снова включить игровой канал.

Каталог `/guest-portal/gamification/clubs` дополнительно возвращает `verification`: ранжированный план каналов входа для `/play`. Приоритет за Telegram-ботом как будущим фундаментом регистрации, уведомлений, рефералок и возврата гостей; звонок пользователя на номер заложен как дешевый fallback; SMS-код остается обязательным резервом; входящий звонок с последними 4 цифрами помечен как поздний резерв из-за менее очевидного UX. Public UI показывает этот порядок и статусы без раскрытия env, секретов, телефонов или внутренних ID.

Fallback-звонок пользователя включается env `GUEST_PORTAL_USER_CALL_ENABLED=true`, `GUEST_PORTAL_USER_CALL_PHONE_NUMBER` и `GUEST_PORTAL_USER_CALL_SECRET`. `/play` создает `POST /guest-portal/:tenantSlug/:storeId/user-call-auth/start` после ввода телефона и согласия, показывает `tel:`-ссылку на настроенный номер и poll-ит `POST /guest-portal/:tenantSlug/:storeId/user-call-auth/status`. Внешний call-provider после входящего звонка подтверждает caller id через `POST /guest-portal/user-call/confirm` с заголовком `x-guest-portal-user-call-secret`; API сверяет только `phoneHash`, активирует или создает отдельный `GuestGameProfile`, выдает обычный `leetplus_guest_token` и не создает общий `Guest` публичной регистрацией.

Четвертый резервный канал `/play` - входящий звонок с последними 4 цифрами. Он включается только после настройки call-provider env `GUEST_PORTAL_INCOMING_CALL_LAST4_ENABLED=true`, `GUEST_PORTAL_INCOMING_CALL_LAST4_ENDPOINT` и `GUEST_PORTAL_INCOMING_CALL_LAST4_TOKEN`; в dev/demo OTP-режиме API возвращает безопасный `devCode` для QA. Frontend вызывает `POST /guest-portal/:tenantSlug/:storeId/incoming-call-last4/start`, затем гость вводит последние 4 цифры номера звонка в `POST /guest-portal/:tenantSlug/:storeId/incoming-call-last4/verify`. Если канал не настроен, top-level start status остается `NOT_CONFIGURED`; если канал настроен, но доставка заблокирована отпиской или provider policy, API возвращает `BLOCKED`, чтобы QA не путал блокировку с отсутствующими env. Успешная проверка создает или активирует отдельный `GuestGameProfile`, ставит гостевой HttpOnly token через web proxy и возвращает только маски, local match и статус без raw phone, provider token или Langame payload. Guest Game Hub показывает готовность этого резерва отдельным readiness-пунктом `INCOMING_CALL_LAST4_AUTH`.

Guest Game Hub на `/guests/gamification` показывает для этого отдельный readiness-пункт `Звонок пользователя для входа`: видны только статусы флага, номера, callback secret, required env и следующий QA-шаг; сам номер, secret, raw phone и Langame payload не выводятся.

Первичный Telegram-вход на `/play` работает через безопасный handoff: web создает одноразовый Telegram auth challenge для выбранного клуба, открывает deep link бота, затем poll-ит `/telegram-auth/status`. Webhook бота принимает `/start lp_...`, переводит challenge в ожидание contact-share, а после Telegram contact подтверждает телефон, активирует или сливает отдельный `GuestGameProfile` по `phoneHash` и позволяет status endpoint выдать обычный `leetplus_guest_token` через HttpOnly cookie. Ответ webhook дополнительно содержит безопасный `reply` payload для внешнего Telegram-адаптера: после `/start` он просит contact-share через кнопку `request_contact`, а после подтверждения убирает клавиатуру; по умолчанию LeetPlus не отправляет сообщение сам, а при включенном `GUEST_GAME_TELEGRAM_WEBHOOK_REPLY_ENABLED=true` отправляет этот reply через Bot API без раскрытия raw chat id и без сохранения raw update. Сырой телефон не возвращается на frontend, общий `Guest` не создается публичной регистрацией, а событие согласия хранит только маску телефона и маску Telegram identity.

После успешного OTP `/play` сразу запускает безопасную проверку `POST /guest-portal/session/langame-match` и показывает гостю/QA итог сверки: найден ли профиль в локальном Langame snapshot, есть ли совпадения только во внешнем `guests/search`, какой статус связки у `GuestGameProfile`, какие источники проверены и сколько игровых наград, событий, доставок и bonus-ledger записей были довязаны к найденному гостю. Даже если Langame нашел внешнего гостя, но локального snapshot еще нет, публичная регистрация остается в `GuestGameProfile` и ждет обычный guest foundation sync, не создавая новый общий `Guest`.

Для каналов, где raw phone не возвращается в браузер, `telegram-auth/status`, `user-call-auth/status` и `otp/verify` дополнительно возвращают безопасный `match`: статус локальной связки с сохраненным Langame snapshot, `linkStatus`, id связанных игровых/гостевых профилей и счетчики backfill по rewards/events/delivery/bonus ledger. Этот `match` не делает live-запрос в Langame, но позволяет `/play` сразу показать, связан ли участник с уже синхронизированным гостем, и гарантирует, что старые profile-only игровые записи довязаны даже если профиль уже имел `guestId`.

Для будущего Telegram/Mini App и легкого игрового виджета добавлен read-only `GET /guest-portal/session/game-summary`: endpoint работает по существующему guest-token, переиспользует текущий `GuestPortalPayload` и возвращает компактное состояние игрока, бонусного баланса, готовых наград, featured-миссий, Battle Pass, next actions и каналов без полного `guestSnapshot`, timeline, raw phone или live-запросов в Langame. `/play` использует этот контракт как resume-gate для активной сессии, а публичный экран `/play/game` использует его как быстрый игровой интерфейс после OTP: показывает профиль, XP, бонусы, готовую награду, ближайшие миссии, Battle Pass и готовность каналов; без guest-token он ведет гостя обратно на `/play`. На готовом игровом экране есть действие `Чекин в клубе`: оно вызывает `POST /guest-portal/session/check-in`, показывает гостю результат или blocker от API и сразу обновляет игровой summary. Featured-миссии в summary дополнительно несут публичные `questSteps`, единицу прогресса и признак ручного подтверждения, поэтому `/play/game` показывает квест как чеклист с текущим шагом, deadline, XP и понятным статусом автоматической или ручной выдачи бонуса. Battle Pass в summary отдает bounded-дорожку ближайших уровней вокруг текущего/следующего прогресса, а `/play/game` показывает XP-порог, free/premium награду и состояние уровня без полного season JSON. Блок `План игры` показывает все ближайшие bounded next actions с приоритетом, статусом, прогрессом и ссылкой к нужному блоку игрового экрана или в гостевой кабинет. Блок `Результат квеста` на этом же экране объединяет готовую награду, последние safe-записи кошелька, bounded `rewards.bonusHistory` до 5 guest-visible записей bonus ledger и текущий бонусный баланс, чтобы гость видел очередь, отправку, подтверждение, проверку и баланс после выполнения без перехода в админский Guest Game Hub. Карточка `Следующий шаг` ведет к конкретному блоку игрового экрана или в гостевой кабинет к связке с Langame, а блок `Каналы` открывает `#communications` в кабинете клуба для согласия и привязки Telegram/MAX. Для объяснения прогресса summary отдает ограниченный `activity.recent` из сохраненной safe-активности, а `/play/game` показывает последние события/XP с типом, описанием, клубом и датой без полного timeline/xpHistory. Лутбоксы в summary передаются как bounded featured-список с последним wallet-result; `/play/game` раскрывает уже сохраненный результат и код кассиру, не создавая новую награду и не выполняя Langame write.

Доска миссий в compact summary ограничена 6 featured-квестами, отсортированными по прогрессу. `/play/game` показывает их как вкладки `Доступные`, `Почти готовы`, `Награда` и `Все`, чтобы гость мог выбрать ближайшее действие или проверить завершенный квест без нового API-запроса, live-запроса в Langame и участия админа.

Для просмотра шире featured-доски `game-summary` дополнительно отдает bounded `missions.history` до 12 safe-квестов с тем же публичным shape: прогресс, шаги, срок, XP, reward label и `rewardStatus`. `/play/game` показывает `Историю квестов` как компактную ленту активных и завершенных состояний без отдельного API-запроса, raw phone, токенов и Langame payload.

Каждая featured-миссия в `game-summary` теперь несет safe `rewardStatus`: награда впереди, квест выполнен, ждет подтверждения, готова, в очереди, отправляется в Langame, начислена, требует проверки, отменена, выдана или сгорела. `/play/game` показывает этот статус прямо в карточке квеста вместе с суммой/label награды, датой события и балансом после подтверждения, а вкладка `Награда` опирается на реальные wallet/ledger-состояния без raw phone, токенов и Langame payload.

В тот же контракт добавлен блок `progress`: `summary` показывает XP, уровень, XP до следующего уровня, выполненные и почти готовые квесты, готовые/ожидающие награды и суммы bonus ledger, а bounded `timeline` объединяет последние safe-события активности, reward wallet и Langame bonus ledger. `/play/game` показывает это как блок `Мой прогресс`, чтобы гость видел, что уже засчитано и что еще ждет обработки, без отдельного запроса, raw phone, токенов и Langame request/response.

`game-summary` также отдает guest-safe блок `journey`: регистрация, связь с Langame, активность/чекин, квест, награда и начисление бонуса. `/play/game` показывает этот маршрут как `Путь к бонусу` с процентом готовности, следующим шагом и ссылками к нужному блоку экрана или в гостевой кабинет к `#langame-match`, чтобы игрок и QA видели полный путь от `/play` до подтвержденного бонуса без live-запросов, raw phone и участия админа.

Для рефералок `GET /guest-portal/session/game-summary` отдает блок `referral`: opaque HMAC-код, ссылку `/play?clubId=...&ref=...`, share-text, подсказку канала и safe `stats` по принятым/eligible регистрациям. `/play/game` показывает карточку `Рефералка` с копированием ссылки, native share/fallback, Telegram share-url и текущим статусом приглашений, а `/play` при входе по `ref` показывает безопасный баннер приглашения. Код строится от отдельного `GuestGameProfile` и клуба, статистика считается из `GuestGameEvent` `GAME_REFERRAL_ACCEPTED` по `inviterProfileId`; raw phone, secret, сырой profile id и общий массив `Guest` не раскрываются и не используются для публичного статуса. Eligible-регистрация по ссылке попадает в Guest Game Hub как `GUEST_GAME_REFERRAL`, а для production можно задать `GUEST_GAME_REFERRAL_SECRET`, иначе используется `JWT_SECRET`/`APP_ENCRYPTION_KEY`.

Внутренний Guest Game Hub на `/guests/gamification` показывает пилотный чек-лист готовности клуба к первому бонусу: 1337 выбирается приоритетно, далее проверяются `/play`, OTP, игровой профиль, связка с Langame, активные правила, наличие сохраненных `guests/logs` raw-типов, тестовое событие, очередь наград, bonus ledger и обязательная сверка баланса после первого production-начисления. Пункт `Факты guests/logs` показывает количество сохраненных логов, типов, источников и следующее действие для `/sync`; пустой каталог остается data-риском для будущих правил и anti-fraud, но не блокирует dry-run, пока активные правила не используют `guestLogTypes`/`blockedGuestLogTypes` или XP `guestLog`. Если активное правило зависит от `guests/logs`, этот пункт становится prerequisite-блокером runbook и ведет оператора на `/sync` до dry-run. Каждый пункт checklist дополнительно может отдавать безопасные `actionHref/actionLabel`, поэтому UI показывает кликабельный следующий шаг к `/stores`, `/play`, гостевому кабинету, `/sync`, правилам или CSV-export каталога без нового API-запроса. Поверх чек-листа есть runbook-стадия `Стоп / Dry-run / Canary / Live write / Сверка / Готово`: она показывает, можно ли сейчас запускать dry-run, canary, live-write или только сверку, и дает следующий безопасный шаг без live-запросов в Langame и без раскрытия ПДн. Runbook также возвращает action-list для оператора: открыть dry-run/process-event, поставить approved bonus-награду в ledger, выполнить ledger dry-run, запустить защищенный canary live dispatch и перейти к сверке баланса; web UI вызывает только существующие guarded handlers и не добавляет отдельный путь записи в Langame. Canary dispatch дополнительно защищен backend-флагом `canary=true`: сервер обрабатывает максимум одну уже подготовленную ledger-запись и не ставит новые approved rewards в очередь внутри live dispatch. Перед live-write runbook выполняет scoped preflight по клубу 1337: считает claimable ledger-записи (`PENDING`, retry-ready `FAILED`, stale `PROCESSING`), показывает безопасный preview первых claim-кандидатов и разблокирует canary только если готова ровно одна запись. Из preview оператор с правом на награды может отменить показанную pending/failed/stale-processing запись через существующий cancel API: backend не дает отменить свежий `PROCESSING` lock и сохраняет каскадную отмену linked reward/delivery. Действия runbook передают `storeId` выбранного пилотного клуба, поэтому постановка в ledger, dry-run preview, status counts и live claim ограничены клубом 1337, а общий batch dispatch в карточке bonus ledger остается без такого scope. Финальный gate `firstBonusReconciliation` отдельно ищет первую подтвержденную положительную `bonus_balance` операцию по клубу 1337, игнорирует денежные `balance` записи и переводит пилот в `Сверка` или `Готово` только после сверки `balanceAfter` с последующим `GuestBonusBalanceSnapshot`.

На той же странице есть управляемый журнал bonus ledger: последние операции начислений показывают статус, сумму, клуб, маску контакта, attempts/next retry, ошибку Langame и сверку confirmed `balanceAfter` с последним сохраненным `GuestBonusBalanceSnapshot`; raw phone, токены и полный Langame payload не выводятся. Отдельный read-only блок сверяет `GuestBonusBalanceCurrent` с последним `GuestBonusBalanceSnapshot` по `guestId` или внешнему Langame-id и показывает `сошлось / расхождение / ждет sync / нет snapshot` после ledger-начислений без live-запросов в Langame. Оператор с правом на награды может поставить approved bonus-награды в ledger, выполнить safe dry-run dispatch и запустить защищенный dispatch; реальная запись в Langame по-прежнему включается только backend-флагом `LANGAME_BONUS_ACCRUAL_ENABLED=true`. Если ledger-запись до подтверждения отменяется, backend не дает отменить свежий `PROCESSING` lock, а pending/failed/stale-processing записи атомарно отменяют связанную approved-награду и непосланные delivery с событием `DELIVERY_CANCELED_BY_LEDGER`, чтобы reward не оставался готовым к выдаче после остановки Langame-начисления.

- Уже подтвержденные группы не предлагаются повторно без изменений.
- Ранее отклоненная конкретная группа не предлагается повторно в том же составе.
- Есть ручной режим парсинга: можно создать сетевой SKU вручную, добавить товары в группу или отредактировать уже существующую группу.

## Правила синхронизации и истории

Эти правила важны, потому что отчеты строятся на исторических фактах продаж и остатков.

- Повторная синхронизация должна быть идемпотентной: уже загруженные операции обновляются/сопоставляются по внешним идентификаторам, а не создают дубли.
- История продаж не должна ломаться при переименовании товара: в фактах продаж хранится снимок названия на момент продажи.
- Если товар пропал из номенклатуры Langame, исторические продажи сохраняются; сам товар может быть помечен как отсутствующий/неактивный через служебные поля.
- Синхронизация из Langame не должна сама проставлять `canonicalProductId`.
- Объединение товаров в канонический SKU выполняется только через утилиту анализа и ручное подтверждение.
- Любые изменения, влияющие на историю, должны сохранять внешние ID, snapshot-поля и связь с tenant/store.
- Будущая история гостей не должна ломаться при изменении профиля гостя в Langame.
- Синхронизация гостей не должна перетирать ручные CRM-статусы, заметки, сегменты, согласия на коммуникации и другие данные, созданные в LeetPlus.
- Синхронизация Langame не должна перетирать задачи персонала, чеклисты, регламенты, обучение и другие операционные данные, созданные в LeetPlus.
- Персональные данные гостей считаются чувствительными: телефон, email, ФИО, дата рождения и документы требуют отдельного контроля доступа и осторожного хранения.
- Для коммуникаций с гостями нужны согласия по каналам, история согласий и возможность отписки.

## Langame

Подключение выполняется через `/settings`:

1. Указать API-ключ.
2. Добавить домены клубов.
3. Сохранить настройки.
4. Запустить синхронизацию.

Домены клубов вводятся одним из двух способов:

- через запятую с пробелом: `1337.langame.ru, 443.langame.ru`;
- каждый домен с новой строки.

В доменах не нужно указывать `https://` и путь `/public_api`; приложение нормализует домен и само формирует базовый URL вида `https://<domain>/public_api`.

Ключ сохраняется на backend в зашифрованном виде и привязывается к текущему `tenant`.

Базовый URL домена:

```text
https://<domain>/public_api
```

Заголовок авторизации:

```text
X-API-KEY
```

Ключевые методы, используемые интеграцией:

- `GET /clubs/list`
- `GET /products/list`
- `GET /goods/list?club_id=...`
- `GET /products/expense?page_limit=&page=&date_from=&date_to=`
- `GET /products/arrival?page_limit=&page=`
- `GET /transactions/list`
- `GET /log_cash_transaction/list`
- `GET /all_operations_log/list`

Если один из вспомогательных endpoints Langame падает, sync job может быть помечен как `FAILED`, даже если уже полученные товары, остатки и продажи сохранены. Это ожидаемое поведение внешней интеграции: статус job отражает полный проход, а не факт сохранения каждой отдельной сущности.

### Плановая синхронизация

Для cron и сервисных вызовов предусмотрен endpoint:

```text
POST /integrations/langame/scheduled/sync
Header: x-sync-service-token: <SYNC_SERVICE_TOKEN>
```

Поддерживаемые режимы:

- `QUICK`
- `INVENTORY`
- `CATALOG`
- `BACKFILL`
- `FULL`

Пример вызова:

```powershell
curl -X POST "https://api.example.ru/integrations/langame/scheduled/sync" `
  -H "x-sync-service-token: <SYNC_SERVICE_TOKEN>" `
  -H "Content-Type: application/json" `
  -d "{\"mode\":\"QUICK\"}"
```

## Модель доступа

`Tenant` - сеть клубов. Пользователь принадлежит одному tenant и видит только данные своей сети.

`Store` - конкретный клуб внутри сети.

Для интеграции по клубам сохраняются внешние идентификаторы и домены, а tenant-уровневые ключи лежат в зашифрованном виде.

## Важные правила

- Не коммитить `.env` и реальные API-ключи.
- Не менять Prisma на `v7` без отдельного решения.
- Не поднимать вторую PostgreSQL в Docker локально без явной необходимости.
- Не менять `APP_ENCRYPTION_KEY` без процедуры ротации.
- Не делать автоматическое объединение товаров во время Langame sync.
- Не удалять исторические факты продаж при переименовании или исчезновении товара из внешней номенклатуры.
- Любые изменения в расчетах прибыли, маржи, ABC и LFL должны явно учитывать источник себестоимости.
